# Selko iOS App

Native iOS companion app for Selko.

## Architecture

The app talks to Supabase directly for client-safe queries and uses the Python API only for server-side operations like OAuth, sync triggers, and LLM-backed processing.

## Tech Stack

- Swift + SwiftUI
- `supabase-swift`
- iOS 26.2+
- Xcode 27+

## Project Structure

```text
ios/
├── Selko/                  # Application code
├── SelkoTests/             # Unit tests
├── SelkoUITests/           # UI and screenshot tests
├── Config/                 # Debug/Release xcconfig files
└── iOS.xcodeproj
```

## Configuration

Runtime config is read from `Info.plist`, which is populated by the xcconfig files in [`Config`](Config):

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`
- `SELKO_API_URL`

Debug builds use committed local defaults:

- `SUPABASE_URL=http://localhost:54321`
- `SELKO_API_URL=http://localhost:8000`
- the local Supabase anon key generated by `supabase start`

Release builds must provide real values at build time. [`Release.xcconfig`](Config/Release.xcconfig) intentionally ships placeholder values, and the `Validate Release Config` build phase fails the build if any required URL or key is still unset.

## Building And Testing

```bash
# Resolve Swift packages
xcodebuild -resolvePackageDependencies -project ios/iOS.xcodeproj -scheme iOS

# Local unit-test lane
xcodebuild test -project ios/iOS.xcodeproj -scheme iOS-Unit \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -resultBundlePath ios/UnitTests.xcresult

# Full UI / XCUITest lane
xcodebuild test -project ios/iOS.xcodeproj -scheme iOS \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -resultBundlePath ios/TestResults.xcresult
```

CI uses `build-for-testing` plus `test-without-building` against the shared `iOS-Unit` scheme so unit tests can run as a dedicated PR gate.
